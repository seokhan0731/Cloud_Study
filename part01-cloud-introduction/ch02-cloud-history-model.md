# 클라우드 컴퓨팅의 역사와 모델

## 2.1 클라우드 컴퓨팅 정의

### NIST의 정의

클라우드 컴퓨팅은 네트워크, 서버, 스토리지 등과 같은 **컴퓨팅 자원**을 설정 가능한 **공유 풀**로 구성하여, 서비스 제공자가 제공하면 서비스 제공자의 **간섭**과 **컴퓨팅 자원 관리**를
**최소화**하면서 언제 어디서나 **필요에 따라** 컴퓨팅 자원을 **네트워크**를 통해 편리하게 접근할 수 있는 모델이다.

* **NIST외 여러 기업 등의 정의가 존재하지만, NIST의 정의가 일반적으로 통용**
* **온디맨드, 확장성, 인터넷, 컴퓨팅 자원 등에 대한 서술이 여러 정의해서 나타남**
* **NIST 포함 여러 정의는 사용자 관점에서 정의된 것**

## 2.2 클라우드 컴퓨팅의 특성과 배포 모델

### 클라우드 컴퓨팅 5가지 특성

NIST가 제시

1. 온디맨드 셀프 서비스

사용자는 제공자의 직접적인 관여 없이 원하는 컴퓨팅 자원을 자동으로 프로비저닝할 수 있다.  
=> 사용자는 필요할 때, 별도 작업없이 사용 가능

2. 광대역 네트워크 접근

네트워크를 통해서 다양한 이기종 클라이언트 플랫폼을 통해 컴퓨팅 자원에 접근할 수 있다.  
=> 네트워크를 통해 다양한 접근 방식을 통해 자원에 접근 가능

3. **리소스 풀링**

컴퓨팅 자원은 풀링되어 멀티테넌트 모델로 다양한 사용자에게 제공된다. 풀링된 자원(프로세싱 파워, 메모리, 네트워크 대역폭, 가상 머신...)은 사용자들의 요구에 맞게 제공되고 회수되어 다른 사용자에게 재할당된다.

**풀링:**  
필요할 때 언제든 꺼내 쓸 수 있도록 자원을 한 곳에 모아둔 거대한 웅덩이(풀)에 자원을 모아두는 행위

**멀티테넌트(멀티테넌시):**

* 여러 사용자들이 동일한 자원을 공유 및 활용
* how? 각자 구축되는 인프라를 중앙 집중화
* 이점  
  인프라 구축 비용 감소, 필요에 따라 대규모 자원 한 번에 활용 가능

    * 서비스 관점: 보다 적은 비용으로 대규모의 자원 활용 가능
    * 자원 관리 관점: 최소의 자원으로 최대의 효과를 볼 수 있는 자원 최적화의 일환

4. 신속한 탄력성

신속하게 프로비저닝하여 배포 가능하며, 사용자의 요구량에 따라 자원 규모 확장 및 축소 가능
=> 사용자에게 언제든지 원하는 만큼의 자원을 제약 없이 제공받는 것처럼 보임

5. 사용량 측정

서비스 제공자와 사용자에게 자원에 대한 제어, 사용량 등이 투명하게 제공

### 클라우드 컴퓨팅 3가지 서비스 모델

* NIST가 제시
* 사용자 관점으로, aaS(as a Service) 수식어가 붙는다.
* 다양한 기술로 모델을 구현할 수 있지만, 가상화 기술이 가장 효과적이고 효율적이다.  
  => 클라우드 컴퓨팅의 가상화 기술이 본질이나 필수적인 요소는 아니지만, 뗼 수 없는 관계

1. Software as a Service(SaaS)

소프트웨어를 광대역 네트워크를 통해 서비스를 받고 사용한 만큼 비용 지불  
=> 사용자는 소프트웨어를 직접 설치하지 않고 인터페이스를 통해 접근하기에 업데이트 등과 같은 관리로부터 자유로워짐

2. Platform as a Service(PaaS)

개발자와 같은 특정 그룹에 중점을 두어, 동일한 개발 환경을 플랫폼으로 만들어 네트워크를 통해 서비스를 받고 사용한 만큼 비용 지불  
=> 사용자는 동일한 개발 환경을 통해, 개발 속도 증가 및 신속한 문제 대응 가능

3. Infrastructure as a Service(IaaS)

물리적 시스템(컴퓨팅 서버, 네트워크 자원, 스토리지 자원)을 네트워크를 통해 시스템 임차하여 사용하고 사용한 만큼 비용 지불  
=> 사용자는 제공된 자원에 필요한 OS, 미들웨어, 애플리케이션 등 직접 제어 및 관리  
=> 서버 구축 비용 대비 저비용으로 필요한 성능 확보 가능  
=> 일반적으로 물리시스템을 직접 제공하기보단, 가상화를 통해 인프라를 가상 자원으로 제공

### 클라우드 컴퓨팅 4가지 배포 모델

* 클라우드 컴퓨팅의 인프라를 어떻게 구축할 것인가 관련  
  => 3가지 서비스 모델을 어떤 인프라로 모델링 할 것인가

1. 프라이빗 클라우드

특정 기관(동일 기관)의 사용자에게만 오픈되고 사용된다.  
일반적으로 해당 기관이 인프라를 직접 소유하고 자체적으로 관리한다.  
=> **클라우드 버스팅에 상대적으로 많은 시간 소요**

2. 커뮤니티 클라우드

특정 목적의 커뮤니티 사용자에게만 오픈되고 사용된다.  
프라이빗 클라우드가 동일 기관으로 제약이 존재한다면, 커뮤니티 클라우드는 서로 다른 기관이더라도 동일 커뮤니티이기만 하면 된다.  
여러 기관이 인프라를 소유하고 관리하기에, 서로 다른 기관의 인프라를 하나의 논리적인 클라우드 인프라로 구성하는 기술과 보안 정책이 반드시 필요하다.

3. 퍼블릭 클라우드

일반 사용자에게 오픈되고 사용된다.  
반드시 비즈니스 목적일 필요는 없으며, 구축하는 주체가 일반적으로는 중요하지 않다.

4. 하이브리드 클라우드

1~3의 클라우드 배포 모델이 혼합하여 클라우드 인프라를 구축하는 것이다.  
=> **프라이빗 클라우드의 클라우드 버스팅 제약 해결**

**클라우드 버스팅 제약 해결 by 하이브리드 클라우드**  
프라이빗 클라우드에 특정 시점 트래픽 폭증/대규모 연산 필요  
=> 내부 자원이 부족해지는 순간, 퍼블릭 클라우드와 묶어 특정 요청 해당 클라우드로 위임(동적 확장)  
=> 상황 종료 후, 자원 반납

**클라우드 버스팅**  
클라우드 자원을 동적으로 확대하는 것으로, 반드시 **로드 밸런싱**이 수반되어야 한다.

**로드 밸런싱**  
네트워크 트래픽을 여러 서버에 균등하게 분산하여, 단일 서버의 과부하 방지  
=> 시스템의 안정성과 가용성 극대화

| 구분  | 프라이빗 클라우드 | 커뮤니티 클라우드 | 퍼블릭 클라우드    | 하이브리드 클라우드 |
|-----|-----------|-----------|-------------|------------|
| 확장성 | 제한적       | 제한적       | 매우 높음       | 매우 높음      |
| 복원성 | 매우 높음     | 매우 높음     | 중간          | 중상         |
| 보안  | 높은 보안 수준  | 준수한 수준    | 서비스 제공자 의존적 | 준수한 수준     |
| 성능  | 좋음        | 매우 좋음     | 중하          | 좋음         |
| 비용  | 매우 높음     | 높음        | 매우 높음       | 높음         |

## 2.3 클라우드를 활용하는 이유

1. 인프라 규모 산정의 용이성

IT 자원은 비용과 직결되기에, 최적의 IT 자원의 규모 산정이 중요한데, 클라우드는 자원이 필요할 때 필요한 만큼만 동적으로 자원을 확보 가능하기에 가장 바람직하다.  
**가상화가 동적 할당을 가능케 함**

2. 비용의 절감

* 동적으로 자원을 할당하고 해제하기에, 불필요한 자원 낭비 최소화를 통한 비용 절감
* 직접 자원 소유 및 관리할 때 드는 부가적 비용(인력, 설비 비용)을 클라우드 업체의 구독 서비스로 전환하여, 유지보수 및 소프트웨어 구입 비용 절감

3. 탄력성

요구량 변화에 맞추어, 자원 동적으로 확대 및 축소 가능  
=> 상황에 맞추어 탄력적으로 요구 반영 가능

4. 가용성

예측되지 않은 서비스 중단 최소화 및 서비스 가용성(인프라 활용 가능 시간 관련) 증가  
=> 신뢰성까지 향상 가능

**인프라 가용성 수식 관련**  
$\text{Availability} = \frac{T_{\text{operation}}}{T_{\text{total}}} = \frac{T_{\text{total}} - T_{\text{shutdown}}}{T_{\text{total}}}$

유지보수와 같은 셧다운 상태를 모두 포함하여, 실제 인프라가 사용 가능한 시간의 비율

$\text{Reliability} = \frac{T_{\text{operation}}}{T_{\text{available}}} = \frac{T_{\text{total}} - T_{\text{shutdown}}}{T_{\text{total}-T_{\text{scheduled-shutdown}}}}$

예측된 서비스 다운 시간 제외한 인프라 가용성

5. 신속성

가변적인 IT 자원 요청에 신속한 대응 가능(by 가상화)  
=> 안정성까지 향상 가능

6. 투명한 사용량 측정

사용량을 측정 가능하기 때문에, 사용자는 예측 가능한 비용을 산정

7. 복원성

문제 상황 발생시, 가상화 되어 있기에, 손쉽게 복원 가능

## 2.4 클라우드 컴퓨팅에 영향을 준 기술들

### 클러스터링

* **정의**  
  여러 자원을 묶어 마치 하나의 거대한 자원으로 활용할 수 있는 기술이며, 대개 동일한 IT 자원을 이용하며, 광대역 네트워크보단, 동일한 물리적 공간에서 연결된다.  
  => **부족한 컴퓨팅 파워를 늘릴 수 있으며, 상황에 따라 자원과 분리해서 활용 가능**

* **클러스터링 vs 그리드**  
  대개 클러스터링과 그리드 모두, 컴퓨팅 자원들을 하나로 묶어 활용성을 높이며,  
  일반적으로 클러스터링은 강한 결합, 그리드는 느슨한 결합으로 묶여있는 것을 의미  
  => 클러스터링(대개 동일한 물리적 공간에서 연결)  
  => 그리드(대개 광대역 네트워크 사용)

* **개념적 구성**
    * 컴퓨팅 자원: 실제로 묶이는 단위이며, 자원간 LAN 네트워크를 통해 통신
    * 제어 스테이션(마스터 노드): 컴퓨팅 자원을 통제하며, 별도의 컴퓨터 또는 일반 컴퓨팅 자원에 마스터 역할을 부여함으로써 동작
    * 스토리지 자원: 노드들이 접근할 수 있는 공용 저장소로, 노드들은 LAN 대신 전용 네트워크인 **SAN**을 통해 접근

* **클러스터링 소프트웨어(HTCondor)**
    * 목적: 네트워크로 연결된 하드웨어적 인프라를 하나의 시스템처럼 통합 관리하기 위함(마스터 노드처럼 소프트웨어적으로 역할 부여)
    * 구성
        * 작업 수행 머신(Executor Machine)  
          실제 작업을 수행하는 컴퓨팅 노드
        * 작업 제출 머신(Submit Machine)  
          사용자로부터 작업을 받아주는 역할을 수행하는 노드로, 큐를 통해 관리
        * 중앙 관리자(Central Manager)  
          마스터 노드 역할로, 작업 제출 머신에서 받은 작업을 가용 워커 노드들을 확인하여, 할당하는 역할을 수행

### 그리드 컴퓨팅

* **정의**  
  분산된 이기종의 컴퓨팅 자원을 광대역 네트워크를 이용하여 하나의 단일 시스템으로 만들어 컴퓨팅 자원을 공유하는 시스템  
  => 그냥 클러스터링을 이기종 범위로 확대한 것이라고 받아들여도 무방할듯

* **특징**  
  서로 다른 데이터센터들이 느슨하게 연결되어 있기 때문에, 각 데이터 센터별 자치권 보장  
  => 작업이 들어오더라도, 기존 데이터센터의 작업에 대해 먼저 우선순위 부여 가능

* **미들 웨어**  
  각 데이터 센터의 가용 자원을 확인하고, 작업이 그리드에 제출되었을 때, 이를 가장 적절한 데이터 센터로 할당  
  =>클러스터링에서의 마스터 노드를 이기종의 더 넓음 범위로 확대했다고 받아들여도 무방  
  =>but 클러스터링은 동일한 자원이고, LAN으로 연결되어 별도 변환이 필요 없지만, 그리드는 이기종이기 떄문에, 서로 다른 데이터 센터간 통신 프로토콜을 표준 그리드 규격으로 변환

* **그리드 인증서**  
  서로 다른 데이터 센터들을 커뮤니티 단위로 묶었을 때, 컴퓨팅 자원은 해당 커뮤니티끼리 공유하기 때문에, 해당 커뮤니티에 속하지 않으면 작업 거부 필요  
  => 그리드 인증서를 통해, 소속 VO(Virtual Organization) 구분 및 작업 수락 여부 결정

### 가상화

* **정의**  
  한 대의 물리 서버를 여러 대의 가상 서버로 만들어서 제공하는 기술  
  => 결국 컴퓨터는 CPU, 메모리, I/O 구성이기에, 이를 소프트웨어적으로 동작하도록 가상 컴퓨터 구현  
  => 하이퍼 바이저를 통해, 하나의 물리 서버에서 이런 가상 서버들을 여러 개를 만들어, 해당 가상 서버들이 동일한 물리 자원 이용하도록 함

* **필요성**  
  데이터 센터의 환경에서 일반적으로 서버 활용률이 낮기에, 자원 효율성이 떨어지며, 낭비가 되고 있음  
  => 하나의 물리 서버를 여러 개의 논리적인 단위로 쪼개서 제공하여, 자원 효율성 향상

## 2.5 클라우드 서비스 제공자의 역할

* **핵심**  
  결국 클라우스 서비스 제공자의 주요 역할에 대한 핵심은 인프라를 직접 구축하고 관리하는 것보다 더 경쟁력이 있어야한다는 것

* **요구 사항**
    * 쉬운 접근성과 높은 보안성  
      네트워크를 통해 접근하기에, 접근이 쉽고 보안성이 높아야한다.
    * 클라우드에 대한 높은 이해  
      결국 클라우드를 활용함으로써, IT 자원을 쉽게 확보하고 직접 구축하는 것에 비해, 리스크가 적기 때문이라는 사실을 바탕으로 클라우드의 속성을 높은 이해가 요구 된다.
    * 약속된 성능 제공  
      결국 클라우드를 활용함으로써, 자기 자신이 원하는 성능을 쉽게 확보하는 것이 목적이기에, 사용자로 하여금 성능 예측과 약속된 성능이 제공되어야한다.
    * 합리적인 비용  
      결국 클라우드를 활용함으로써, 직접 인프라를 구축하는 것보다 비용적인 이점이 존재해야하기에, 가격은 합리적이여야 한다.
    * 온디맨드 기반의 서비스 제공  
      결국 클라우드를 활용함으로써, 언제든이 사용자의 입맛에 맞게 자원의 크기를 확대/축소할 수 있어야 한다. 이 과정에서 네트워크를 통해 접근할 때, 다양한 종류의 디바이스를 통해서 서비스를 제공받을 수
      있어야한다.
    * 유지보수 부담 제거  
      결국 클라우드를 활용함으로써, 유지보수에 대한 비용을 줄이는 것이 목적이기에, 사용자에게 유지보수의 부담을 안기면 안 된다.

## 2.6 자원성능의 확장

* **분류**
    * 수평적 확장: 스케일 아웃
    * 수직적 확장: 스케일업

### 스케일 아웃

동일한 타입의 IT 자원을 물리머신의 풀에서 제공 가능한 규모만큼 추가하여 확장

### 스케일 업

가상 머신 자체의 성능을 물리시스템에서 제공하는 한계까지 향상  
(Ex. CPU 2개 -> CPU 4개)

### 스케일 아웃 vs 스케일업

확장 용이성? 간편성?  
=>스케일 아웃이 훨씬 유리

만약 기존 물리 서버로 감당 가능하지 않을만큼 트래픽이 늘어나는 경우, 스케일 아웃 방식에서는 물리 서버를 붙인 후, 거기서 또 가상 머신 확장하면 문제 X  
=>무중단 가능  
but 스케일업 방식에서는 기존에 쓰던 가상 컴퓨터요소(CPU, 메모리...)를 통째로 바꿔야 한다.  
=>중단 필요

| 구분    | 스케일아웃            | 스케일업              |
|-------|------------------|-------------------|
| 서버    | 저비용              | 고비용(특화)           |
| 가용성   | IT 자원 즉시 이용 가능   | IT 자원 즉시 이용 어려움   |
| 확장성   | 자원 복제 및 자동 확장 수월 | 추가 설정 필요          |
| 추가 서버 | 추가 IT 서버 필요      | 추가 IT 서버 필요 X     |
| 용량 제한 | 하드웨어 용량에 제한받지 않음 | 최대 하드웨어 용량에 제약 받음 |

## 2.7 클라우드를 사용할 때 고려할 사항

* **핵심**  
  결국 네트워크를 통해 접근하며, 인프라 구축 및 관리를 위탁하기 때문에, 아래와 같은 사항들을 고려해야한다는 것
* **고려 사항**
    * 데이터에 대한 보안 취약성  
      제3자가 관리하기에 보안이 취약해질 수도 있다.
    * 보안정책의 상충  
      독자적인 보안정책과 서비스 제공자의 보안정책이 상충할 수도 있다.
    * 어드민의 역할 축소  
      직접 인프라를 구축하는 것보다, 당연히 어드민의 역할이 축소되며, 클라우드 서비스 제공자에 대한 의존성이 커진다.
    * 네트워크 의존성 증가  
      네트워크가 필연적이기에, 기본적인 네트워크의 취약성을 갖게 된다.
    * 클라우드 간의 이식 표준 부재  
      클라우드 서비스 업체 변경시, 이식 표준이 존재하지 않아, 동일한 환경 보장 X
    * 지역적 이슈  
      네트워크를 통해, 사용자의 인프라가 타 대륙, 타 국가에 존재할 수 있기 때문에, 해당 위치의 국가 정책 등이 적용될 수 있다.
