# 클라우드 컴퓨팅 개념

## 1.1  클라우드 관련 용어

* **클라우드 컴퓨팅:** 온디맨드 방식으로, 원하는 시점에 원하는 컴퓨팅 리소스를 인터넷을 통해 접근하여 사용할 수 있게 제공하는 행위이다.  
  이때, 인터넷을 통해 접근하기에, 사용자는 이 존재를 인지할 수 있지만, 그 실체를 직접 확인할 수는 없다.
* **온디맨드:** 원하는 시점에, 원하는 리소스 사용 가능 -> 유연성
* **데이터 센터:** 인터넷을 통해 리소스 제공하는 주체

## 1.2 클러스터링

* **정의:** 독립된 컴퓨터를 네트워크로 연결 -> 하나의 컴퓨터처럼 활용 가능(단일 시스템)
* **연결 강도에 따른 분류**
    * 느슨한 연결: 서로 다른 기관 또는 데이터 센터 -> 각 컴퓨터는 각 센터 정책에 따라 제약 O
    * 강력한 연결: 동일한 데이터 센터 내 -> 각 컴퓨터는 동일한 정책 -> 제약 X


* **왜 필요?**
    * 결국 컴퓨팅 성능은 하드웨어(CPU, 메모리)와 밀접한 연관 O  
      but CPU의 경우는 발열에 따른 쿨링이 필수적 -> 비용 문제 발생    
      메모리의 경우는 메모리 뱅크(장착 가능한 메모리의 개수) 정해짐 -> 메모리 무한대로 높일 수 X
    * => 클러스터링 필요(논리적 컴퓨터), 이를 통해 사용자는 하나의 컴퓨터처럼 활용 하며, 내부에서는 작업을 연결된 컴퓨터들이 병렬로 처리

## 1.3 컴퓨팅 자원의 효율적 활용

* **기본 전제:**   
  컴퓨팅 자원이 어떻게 쓰이고 있냐는 배제하고, 컴퓨팅 자원이 어떤 작업을 위해 사용된다면, 효율적 활용 중


* **방법론적 분류 => 어떤 컴퓨터를 끌어 쓸건가**
    * 특정 시간동안 사용되지 않은 컴퓨팅 자원 활용(시간성 측면): 특정 시간동안 유휴한 컴퓨터 끌어 쓰기,  
      이때 여러 컴퓨터가 현재 유휴한지 직접 확인하는 것은 시간 소모적  
      => 중재자 필요: 중재자가 요청 시점에 유휴자원 확인 후, 자원을 할당하여 작업 처리하도록  
      ex. SETI@HOME
    * 작업의 부하가 작은 컴퓨팅 자원 활용(활용성 측면): 특정 상황에서, 컴퓨팅 리소스를 조금만 사용하는 컴퓨터 끌어 쓰기,  
      이때도 중재자가 필요(마찬가지로, 얼마나 리소스가 남는지를 직접 확인하는 것은 소모적 작업이기에)


* **컴퓨팅 자원의 활용과 안정성**
    * **Y?** 결국 효율적 활용이 목적인데, 컴퓨팅 환경 등의 차이로 인해 안정성이 보장되지 않는다면, 오히려 작업완료의 시간을 더 늘리는 상황 야기  
      ex. A의 작업이 B로 전송되어 실행되는 중, 시스템 전체에 영향을 주는 작업 수행시, 최악의 경우 B가 다시 재부팅되고 이전에 수행했던 작업을
      처음부터 재시작해야 하는 상황 발생 가능 => A와 B 모두의 처리시간 증가
    * **How to?** 하이퍼바이저 도입  
      결국 A의 작업과 B의 작업을 분리하면 되기에, 물리적 자원을 여러 대의 가상 자원으로 관리  
      => 가상 컴퓨터에서 수행되는 작업이 문제를 일으키더라도, 오로지 해당 가상컴퓨터에만 귀속


* **클러스터 vs 클라우드**
    * 둘 다, 컴퓨팅 자원의 효율적 활용이 목적(유휴한 자원을 최대한 활용하여 많은 양 처리)  
      => HTC(High Throughput Computing) <=> HPC(High Performance Computing)
    * but 그 구현하는 방법론에 대한 차이 존재
        * 클러스터/그리드: 여러 대의 컴퓨터를 네트워크를 통해 하나로 묶는다 => 오케스트레이션 기법
        * 클라우드 컴퓨팅: 하나의 컴퓨터를 여러 대의 논리적으로 쪼갠다 => 가상화(하이퍼바이저 기반 기술) 기법
            * **클라우드 컴퓨팅**의 기반 기술인 가상화를 통해, 데이터 센터 운영 3대 이슈 해결 가능


* **데이터 센터 운영 3대 이슈**
    * 자원의 활용성: HTC 범주이기에 해결 가능
    * 안정성: 여러 개의 논리적 단위로 쪼개고, 문제 발생시, 해당 논리 단위에 귀속되기에
    * 빠른 복구: 복구 필요시, 논리적 단위에서의 **이미지**가 존재하기에, 단순 해당 이미지 이동 관리를 통한 복구 가능


* **클라우드 컴퓨팅 서비스**
    * 분류
        * IaaS(Infrastructure as a Service): 가상화 기능이 탑재된 물리적 머신 제공  
          => 사용자가 직접 가상 서버 생성 및 관리
        * PaaS(Platform as a Service): IaaS 기반으로, 제공자가 가상 서버 생성 및 관리  
          => 사용자는 플랫폼 위에서 동작하는 애플리케이션 및 데이터만 관리
        * SaaS(Software as a Service): PaaS 기반으로, 제공자가 애플리케이션까지 모두 관리
    * **모든 서비스는 사용자가 사용한만큼 비용 청구**

## 1.4 가상화 / 하이퍼바이저 분류

### 가상화

* **게스트 OS의 수정 유무에 따른 분류**
    * 전가상화: 게스트 OS가 수정되지 않는 상태로 동작  
      => 게스트 OS 내부 코드나 커널을 실행환경에 맞추어 수정 X  
      => 게스트 OS는 자신이 가상화 환경에서 동작한다는 것을 인지 X  
      => 게스트 OS가 I/O 요청 등에 대한 민감한 요청시, 하이퍼바이저가 이를 가로채어 중재하고, 필요한 경우 실제 하드웨어 또는 하위 시스템에 전달  
      => **하이퍼바이저가 중간에 가로채어 중재해야 하기 때문에, 오버헤드 존재**
      <br><br>
    * 반가상화: 게스트 OS가 수정된 상태로 동작  
      => 게스트 OS는 자신이 가상화 환경에서 동작한다는 것을 인지    
      => 하이퍼바이저의 중재가 필요한 경우에, 유기적으로 하이퍼바이저와 협력 가능  
      => **하이퍼바이저가 중간에 가로채고, 해석 필요 X (오버헤드 감소)**  
      => 이때. 게스트 OS가 하이퍼바이저에게 하는 작업 요청을 **하이퍼콜**이라고 한다.

### 하이퍼바이저

* **가상화의 분류와는 상관 없이, 가상화를 통해 구현한 것이 하이퍼바이저**
* **해당 하이퍼바이저가 어디 위에 올라가냐에 따른 분류**
    * 타입 1(베어메탈, 네이티브)  
      => 물리시스템 위에 설치  
      => 물리시스템의 자원의 관리를 하이퍼바이저가 직접 가능  
      => 호스트 OS를 거치지 않고 다이렉트로 물리시스템에 접근 가능하니, 성능적으로 우수
      <br><br>
    * 타입 2(호스티드)  
      => 호스트 OS 위에 설치  
      => 물리시스템의 자원 관리를 위해 호스트 OS에 먼저 접근 필요  
      => 성능적으로는 타입 1에 비해 저조 but 사용 편의 등에서 우수

### 하이퍼콜

* 응용 프로그램이 OS에게 요청하는 것을 시스템콜이라고 하는 것처럼, 하이퍼콜은 게스트 OS가 하이퍼바이저에게 요청하는 것을 의미한다.
* 시스템콜과 마찬가지로, 인터페이스가 존재하며, 게스트 OS는 해당 인터페이스에 맞는 형식으로 하이퍼바이저에게 요청을 보내고,
* 하이퍼바이저는 해당 하이퍼콜로부터 정해진 형식에 맞추어 처리한다. (해석 비용 X <=> 전가상화)

### 가상 머신 관리 툴

* 필요성: 하이퍼바이저는 물리 시스템당 필요하기 때문에, 수많은 물리 시스템을 가진 데이터센터의 경우, 해당 하이퍼바이저를 각각 관리해줘야 한다.  
  => 개별 하이퍼바이저들을 각각 관리하는 것은 비효율적이고, 번거롭기 때문에, 하이퍼바이저들을 관리할 수 있는 상위 툴이 필요하다(오케스트레이션 툴).  
  Ex. 오픈스택, 오픈네뷸라

### 그리드 컴퓨팅

* 특징: 느슨한 연결 (<=>클러스터링)
  => 리소스 오케스트레이션 필요
* **리소스 오케스트레이션**
    * 지역적으로 분산되어 있는 자원을 효율적으로 활용하기 위해, 요청된 작업의 속성과 요구사항을 분석하여 데이터 센터의 컴퓨팅 자원과 매칭
    * 자원 활용 중, 지원 요청이 들어오는 경우, 기존 작업들을 종료하거나 우선순위가 뒤로 밀려나고 지원 업무를 우선적으로 처리하는 등의
      스케줄링 및 자원 재할당 수행

## 연습문제

1. 클라우드 컴퓨팅에 대한 정의를 기술해보라.
   <br><br>
   온디맨드 방식으로, 원하는 시점에 원하는 컴퓨팅 리소스를 인터넷을 통해 접근하여 사용할 수 있게 제공하는 행위이다.  
   이때, 인터넷을 통해 접근하기에, 사용자는 이 존재를 인지할 수 있지만, 그 실체를 직접 확인할 수는 없다.


2. 컴퓨팅의 성능, 용량을 늘리는 방법들은 무엇이 있는가?
   <br><br>
   CPU의 클록 스피드를 향상시키거나, 메모리를 더 부착하여 성능과 용량을 늘릴 수 있다.  
   CPU는 발열 해결을 위한 비용 문제가 있고, 메모리는 부착할 수 있는 메모리 뱅크의 수가 정해져있기에, 현실적으로 무제한으로 성능을 늘리는 것은 한계가 있다.


3. 클라우드 컴퓨팅과 그리드 컴퓨팅의 공통점은 무엇이고 차이점은 무엇인가?
   <br><br>
   두 컴퓨팅의 공통점은 모두 HTC 범주에 속하며, 모두 컴퓨팅 자원을 효율적으로 활용하여 많은 작업을 처리하려는 목적을 가진다는 점이다.  
   그리드 컴퓨팅은 지역적으로 분산된 여러 컴퓨팅 자원들을 네트워크로 느슨하게 연결하여 자원을 공동 활용하는 방식으로,  
   분산된 자원을 효율적으로 연결하고 배분하기 위해 리소스 오케스트레이션이 필요하다.  
   클라우드 컴퓨팅은 데이터센터 내부의 물리 자원을 가상화하여, 하나의 물리 자원을 여러 개의 논리 자원으로 분할하여 제공하며, 이를 위해 하이퍼바이저가 필요하다.


4. 자원의 활용성을 높이기 위한 방법에 대해서 설명하라
   <br><br>
   자원의 활용성을 높인다는 것은, 컴퓨팅 자원의 유휴 시간을 줄이고 자원을 효율적으로 배분되도록 하는 것을 의미한다.  
   이를 위한 방법으로는 첫째, 특정 시간동안 사용되지 않는 유휴자원을 활용하는 방법이다. 어떤 컴퓨터가 일정 시간 동안 사용되지 않는다면, 그 자원을 다른 작업 처리에 활용할 수 있다.  
   둘째, 부하가 작은 컴퓨팅 자원을 활용하는 방법이다. 완전히 놀고 있지는 않더라도, 현재 사용량이 낮은 컴퓨터의 남는 자원을 추가 작업에 활용한다.  
   첫째, 둘째 모두 사용자가 직접 확인하는 것은 시간 소모적이고 비효율적이기에, 이를 대신 판단하고 자원을 배분하는 중재자가 필요하다.


5. 가상화와 오케스트레이션에 대해 설명하라.
   <br><br>
   가상화란 하나의 물리적 자원을 여러 개의 논리적 자원처럼 나누어 사용하는 기술이며, 이에 대한 구현은 하이퍼바이저가 된다.
   오케스트레이션은 여러 컴퓨팅 자원을 전체적으로 조정하고 배치하는 관리 방식이다. 요청된 작업의 특성과 요구사항을 분석한 뒤 적절한 자원과 매칭하고,  
   우선순위에 따라 작업을 배치하는 등의 역할을 뜻한다.  
   즉, 가상화는 하나의 물리적 자원을 논리적으로 나누는 기술이고, 오케스트레이션은 여러 자원을 전체적으로 관리하는 기술이다.


6. HTC와 HPC의 차이점을 설명하라.
   <br><br>
   HTC는 High Throughput Computing으로, 일정 시간동안 얼마나 많은 양의 작업을 처리할 수 있는가에 초점을 맞춘다.
   HPC는 High Performance Computing으로, 하나의 큰 작업을 가능한 빠르게 처리하는 성능적인 측면에 초점을 맞춘다.


7. 아마존과 클라우드의 관계에 대해 설명하라.
   <br><br>
   아마존은 4분기에 트랜잭션이 집중되는 특성을 갖고 있는데, 4분기에 트랜잭션의 수요에 맞게 컴퓨팅 파워를 맞춘다면 1~3분기동안 컴퓨팅 파워의 낭비가 발생한다.  
   이렇게 낭비되는 자원은 아마존 입장에서 큰 손실이기에, 잉여 컴퓨팅 자원을 외부 사용자에게 서비스 형태로 판매함으로써 일거양득의 효과를 볼 수 있었고,  
   이러한 방식이 클라우드 컴퓨팅 서비스의 발전으로 이어졌다.


8. 컴퓨터를 가상화할 때 크게 3개의 범주는 무엇이고 각각 가상화는 어떻게 되는지 조사하라.
   <br><br>
   CPU, 메모리, I/O 가상화가 해당 3개의 범주에 속한다.  
   CPU 가상화는 하나의 물리 CPU를 여러 가상 머신이 나누어 사용할 수 있도록 하는 방식이며, 각 가상머신은 자신이 독립적인 CPU를 할당받은 것처럼 인식하지만, 실제로는 하이퍼바이저가 CPU 실행 시간을
   분배하여 여러 가상머신이 번갈아 사용하도록 한다.  
   메모리 가상화도 마찬가지로, 하나의 물리 메모리는 여러 가상 머신이 독립적으로 사용하는 것처럼 보이게 하는 방식이다. 하이퍼바이저가 물리 메모리를 분할하고 주소 변환을 관리한다.  
   I/O 가상화는 디스크, 키보드 등 여러 I/O 장치를 여러 가상 머신이 사용할 수 있도록 하는 방식이다. 마찬가지로 하이퍼바이저가 가상 장치를 제공하고, 이를 실제 물리 장치와 연결한다.


9. 그리드 컴퓨팅의 개념에 대해서 설명하라.
   <br><br>
   그리드 컴퓨팅은 지역적으로 분산되어 있는 여러 컴퓨팅 자원을 네트워크를 통해 느슨하게 연결하고, 이를 활용하여 작업을 처리하는 방식이다.  
   느슨하게 연결돼 있기에, 서로 다른 환경에 속한 컴퓨터들이 각자의 정책에 적용을 받으며, 필요에 따라 협력적으로 자원을 제공한다.  
   다만 서로 다른 환경이기에, 리소스 오케스트레이션이 중요하다. 리소스 오케스트레이션은 작업의 속성과 요구사항을 분석한 뒤 적절한 자원과 매칭하고,
   자원 활용 중 새로운 요청이 들어오면 기존 작업의 우선순위를 조정하거나 일부 작업을 중단시키는 등의 분산 자원을 협력적으로 운영하기 위한 관리 체계이다.


10. HTC, HPC, 그리드 컴퓨팅, 클라우드 컴퓨팅, 활용성, 오케스트레이션, 가상화, 슈퍼컴퓨터와 같은 용어를 논리적으로 연계하여 설명하라.
    <br><br>
    컴퓨팅 환경에서 주된 목표 중 하나는 한정된 자원을 얼마나 효율적으로 활용하냐는 것이다. 이에 대한 두 가지 관점으로 HTC와 HPC가 있는데,   
    HTC는 하나의 작업을 매우 빠르게 처리하는 것에 초점을 맞추고, HPC는 일정 시간동안 얼마나 많은 양의 작업을 처리할 수 있는가에 초점을 맞춘다.
    그리드 컴퓨팅이 HTC 범주에 속하는데, 일정 시간 동안 많은 작업을 처리하고자, 여러 컴퓨터 자원을 연결하여 활용하는 방식이다.  
    지역적으로 분산된 여러 자원을 느슨하게 연결하며, 이 과정에서 작업의 요구사항과 자원의 상태를 분석하여 적절히 배치하는 오케스트레이션이 필요하다.
    한편 클라우드 컴퓨팅도 HTC 범주에 속한다. 그리드 컴퓨팅이 여러 대의 물리적 컴퓨터를 연결하여 활용한다면, 클라우드 컴퓨팅은 하나의 물리적 자원을 여러 개의 논리적 자원으로 분할하여, 활용성을 높인다.  
    이는 가상화를 통해 이루어지는데, 가상화를 통해 하나의 물리 서버를 여러 개의 가상 머신으로 나눌 수 있으며, 이는 단순한 활용성 향상뿐 아니라 안정성과 복구 용이성도 함께 제공한다.  
    반면 HPC는 연산 속도 등의 능력이 중요하며, 이 범주에 속하는 대표적인 예는 슈퍼컴퓨터다. 슈퍼컴퓨터는 매우 높은 연산 성능을 통해 대규모 연산을 빠르게 수행하기 위해 사용된다.


11. 데이터 센터 운영의 3대 이슈를 설명하고 클라우드 컴퓨팅을 통해서 해결할 수 있는지 논하라.
    <br><br>
    3대 이슈는 자원의 활용성, 안정성, 빠른 복구이다.  
    자원의 활용성은 컴퓨팅 자원이 최대한 유휴 없이 효율적으로 사용되는가의 문제이고, 안정성은 하나의 작업이 전체 시스템에 미치는 영향을 최소화하는 것이다.   
    빠른 복구는 문제 발생 시 서비스를 원래 상태로 신속하게 되돌릴 수 있는지를 의미한다.
    클라우드 컴퓨팅은 가상화를 통해 하나의 물리 자원을 여러 개의 논리 자원으로 나누어 자원의 활용성을 높일 수 있고, 가상 머신의 작업과 장애는 해당 가상 머신에 귀속되기에, 안정성을 높일 수 있다.  
    또한 이미지를 이용하기에 빠른 복구도 가능하다.


12. 클라우드 컴퓨팅에 대한 다양한 정의가 존재한다. 클라우드 컴퓨팅을 정의할 때 가상화 기술은 필수적인가? 이에 대해 논하라.
    <br><br>
    필수적이지 않다고 생각한다. 결국 클라우드 컴퓨팅이란, 온디맨드 방식으로, 원하는 시점에 원하는 리소스를 인터넷을 통해 접근, 사용하게 하는 기술이다.  
    가상화는 이를 지원해주는 기술이지, 본질적인 정의를 생각했을 때는 필수는 아니라고 생각한다.   
    가상화 없이 물리 서버 하나를 통째로 빌려주는 것도 클라우드 서비스에 속하기 때문에, 결론적으로 가상화는 클라우드 컴퓨팅을 지원하는 가장 이상적인 기술이지만,   
    기술 자체가 필수적이라고는 생각하지 않는다.


13. 클라우드 컴퓨팅을 자원 활용의 관점에서 보는 것은 지엽적인가? 지엽적이라면 활용성 관점이 아닌 다른 관점에서 클라우드 컴퓨팅의 응용은 무엇이 있는가?
    <br><br>
    지엽적이라고 생각한다. 자원 활용의 관점이 클라우드의 기술적 토대를 설명하기에는 충분하지만, 클라우드가 실제로 무엇에 응용되고 설명하기에는 충분하지 않기 때문이다.  
    응용은 활용성을 넘어 경제 모델의 전환(소유 -> 임대), 안정성 측면, 서비스 측면(IaaS/PaaS/SaaS)으로 볼 수 있다.  
    경제 모델의 전환에 따라, 사용자는 필요한 만큼, 사용한 만큼만 비용을 지불할 수 있게 되었다.  
    또한 장애 격리와 이미지를 통해, 단순히 자원의 활용하는 것과 별개로, 서비스의 안정성, 신뢰성을 보장할 수 있게 되었다.  
    마지막으로 IaaS/PaaS/SaaS 등을 서비스로 운영함에 따라, 사용자로 하여금, 관심사를 채택하여, 다른 관심사와 책임은 위임할 수 있게 되었다.  
    결국, 실제 응용과 가치는 이러한 측면에서 비로소 설명될 수 있다.
